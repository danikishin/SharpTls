# RFC 9846 compliance checklist

RFC 9846 is the project's normative TLS 1.3 specification. This checklist turns the concrete changes enumerated by RFC 9846 §1.2 into implementation and test obligations.

| RFC 9846 change | SharpTls obligation | Verification gate |
|---|---|---|
| KeyShare reuse is forbidden | Each `CustomTlsClient.ConnectAsync` creates fresh key-share owners. An HRR replaces the first share with a fresh share. No profile may inject reusable private key material. | Reference-identity/lifetime test and two-connection public-share inequality test. |
| TLS 1.0/1.1 negotiation forbidden | Executable policies advertise and accept only TLS 1.3 and the separately hardened TLS 1.2 subset. Legacy record/ClientHello versions are compatibility fields, never negotiated TLS 1.0/1.1 versions. | Mixed-version dispatch, downgrade-sentinel, and unoffered-version negative tests. |
| PSK/HRR hash ambiguity removed | Resumption binder hashing uses the ticket suite's hash and reconstructs `message_hash(ClientHello1) || HRR || truncated ClientHello2`; the server-selected suite must remain PSK-hash compatible. | RFC 8448 binder vector, deterministic HRR binder test, and public HRR resumption. |
| Ignore NewSessionTicket when resumption unsupported | Every `NewSessionTicket` is strictly parsed and bounded. Without a caller cache it is ignored; with a cache its PSK is derived and origin/authentication lifetime is bound before storage. | Parser corruption tests, no-cache path, cache rotation, and public resumption. |
| Key-update-before-limit is mandatory | Client write keys rotate automatically before the conservative AEAD record limit; peer and caller-requested updates ratchet secrets and replace/zeroize record ciphers. | Boundary tests, crossed-update tests, and public HTTP traffic after KeyUpdate. |
| Limit permitted KeyUpdate epochs | The sending epoch is a 64-bit counter capped at `2^48-1`. Local/manual updates fail before overflow, while a peer `request_update` is ignored when no legal response epoch remains. Receiving does not incorrectly impose the sender-only epoch cap. | Exact boundary unit test and a multi-epoch protected-record loopback test. |
| Bound KeyUpdate request loops | Repeated peer requests while the client is silent are coalesced before the next application record. A second local `request_update=1` is rejected until a peer KeyUpdate is authenticated. | Managed crossed-update/coalescing tests and outstanding-request negative test. |
| `close_notify` is warning level | Outgoing close uses warning; incoming close accepts the specified level and makes subsequent data unreadable. | Alert wire/good-close/truncation tests. |
| `user_canceled` semantics | A received advisory `user_canceled` is ignored in plaintext, encrypted-handshake, and application states while normal authentication/truncation rules continue to apply. API cancellation closes the transport and is never reported as successful TLS shutdown; sending an advisory alert on local API cancellation is not claimed. | Alert parser tests and cancellation/EOF tests. |
| `general_error` alert | Wire value 117 is modeled and accepted; local code chooses a more specific alert whenever possible. | Enum/wire parser tests. |
| CertificateRequest requires `signature_algorithms` | Initial and post-handshake requests reject the missing extension, malformed/duplicate extensions, illegal contexts, and context reuse. Compatible RSA/ECDSA credentials produce Certificate and CertificateVerify; no match produces the required empty Certificate response. | Parser negatives, RSA/ECDSA signature tests, TLS 1.2/1.3 mutual-auth interop, and PHA hostile loopback tests. |
| Post-handshake authentication is explicit | The empty `post_handshake_auth` ClientHello extension is opt-in. PHA forks the initial transcript through client Finished, adds only the current request/response, uses the current client application traffic secret for Finished, and emits each response consecutively. | Positive, empty-credential, unadvertised-request, and reused-context managed interop tests. |
| Correct ClientHello extension lower bound | Encoder validates actual required extensions and structural length rather than retaining RFC 8446's erroneous presentation bound. | Minimal valid TLS 1.3 ClientHello test. |
| Correct NewSessionTicket extension upper bound | The ticket parser consumes the complete 16-bit extension vector under the enclosing handshake limit, detects duplicate `early_data`, and rejects malformed extension bodies. | Maximum/truncated extension-vector and duplicate-extension tests. |
| Generic asymmetric key-exchange terminology | Core state and key-schedule APIs use `key exchange`/`shared secret`; curve-specific code stays behind `IKeyShare`. | Architecture review. |
| Future transcript extensions clarified | Only explicit Handshake messages enter `TranscriptHash`; future messages require an enumerated transcript action and state transition. | Transcript golden tests and unknown-message negatives. |
| Mandatory RSA-PSS text removed / RFC 9963 alignment | Public profiles may offer the supported RSA-PSS and ECDSA schemes requested by this project. The implementation does not assert that RSA-PSS is universally mandatory and never negotiates legacy PKCS #1 code points in TLS 1.3. | Offered/selected scheme matrix tests. |
| Privacy/security text updated | Deterministic mode remains explicit test-only behavior; GREASE and padding are configurable; logs never contain secrets by default. | API review and secret-log scan. |

The checklist is part of the release gate. A row may move to a feature-specific future milestone, but it may not be silently ignored or represented as implemented.
